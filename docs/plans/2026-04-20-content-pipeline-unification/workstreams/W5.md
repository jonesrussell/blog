# W5 — Intake Unification

## Objective

Fold existing intakes (git-activity miner in `content-mine.yml`, coforge
Redis subscriber code) into the new envelope pattern so every intake
produces a `stage:mined` issue with the same shape as W2's subscriber
emits.

## Repos touched

- `~/dev/blog` — `content-mine.yml`, any blog-side NC subscriber code
- `~/dev/north-cloud` — adjust the coforge emission path if it lives there

## Deliverables

- git-activity miner rewritten to produce NC-shaped envelopes emitted on
  a synthetic `source:git` **Redis** channel consumed by W2's subscriber.
  (In-process is explicitly off the table — consistency with NC's pattern,
  Redis is cheap locally.)
- Legacy coforge Redis subscriber code removed. W2's deploy already
  disabled the subscriber (per W2's Rollout section); W5 deletes the dead
  code after verification.
- Dual-write period: old and new paths emit side-by-side for one week.
  After parity is confirmed, legacy paths removed.
- Blog `CLAUDE.md` documents "one intake shape, one lifecycle" as an
  ongoing rule.

## Acceptance criteria

- All three v1 intake sources (git activity, NC Redis, weekly roundup)
  produce identically-shaped `stage:mined` issues.
- No references to the legacy content-mine issue shape remain in the repo.
- Removing any one intake source does not affect the other two.
- Dual-write parity verified on a sample of issues before cutover.

## Dependencies

- W1a (envelope schema).
- W2 (subscriber + issue shape). W2 already handled the coforge subscriber
  disable at its deploy; W5 handles code removal and the content-mine
  migration.

## Complexity

M — mostly migration work. Risk is in the dual-write period and in
spotting hidden consumers of the old shape.
