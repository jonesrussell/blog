# W3 — Weekly Industry Roundup (Consumer-Side Synthesis)

## Objective

Move the Cowork-scheduled Monday industry roundup into the pipeline as a
consumer-side synthesis job. Read the last 7 days of NC envelopes from the
blog's local index (W2), synthesize the four-section roundup, and emit a
content-queue issue with the roundup attached as the content seed.

## Repos touched

- `~/dev/blog` — new scheduled synthesis job (GitHub Action preferred)
- `~/dev/north-cloud` — only if topic coverage verification requires
  adding feeds to the source registry (see below)

## Pre-execution: topic coverage verification

Before scheduling any synthesis, inventory NC's `source-manager` registry
against the roundup's four topic areas:

1. AI / ML
2. Software development & tooling
3. Tech business & startups
4. Cybersecurity

The roundup is consumer-side synthesis from the blog's local envelope
index. If NC does not crawl sources covering a topic area, the local
index has nothing to feed that section of the roundup.

Coverage outcomes:

- **Full coverage (≥ 2 active feeds per topic area):** proceed with
  consumer-side synthesis only.
- **Sparse coverage (one or zero feeds for any area):** W3's scope
  expands. Pick one:
  - (a) Extend NC's source registry with feeds for the gap areas before
    synthesis runs. This becomes an NC-side task inside W3 and pushes
    `~/dev/north-cloud` into W3's repos-touched list. Document each new
    feed via the standard source-manager workflow.
  - (b) Keep a web-search fallback for topics with no local coverage —
    matching how the Cowork task works today. Document the fallback
    rules and include them in the synthesis pipeline with clear
    provenance labels (envelope-sourced vs. web-search fallback).

Escalate to Russell if the coverage gap is broad enough that both (a) and
(b) would be needed. Do not silently skip topic areas.

## Deliverables

- Topic-coverage inventory document produced during pre-execution, stored
  in this workstream's directory.
- Scheduled job running Monday ~8am UTC that:
  - queries the local envelope index for items in the previous 7 days
    across the four topic areas,
  - synthesizes the roundup using the Cowork task's instruction set
    (preserved verbatim for the first two iterations, then editable),
  - writes the roundup markdown to
    `content/posts/general/industry-roundup-YYYY-MM-DD/index.md` as a
    draft blog post, or to `~/brand/substack-issue-N.md` if the operator
    chose Substack (decide at kickoff; default Hugo draft),
  - opens a **`stage:mined`** content-queue issue with appropriate labels.
    Human curation promotes to `stage:curated` / `stage:ready` per the
    standard lifecycle — pre-curated weekly content still benefits from
    a human sanity-check pass before it advances.
- Two weeks of side-by-side parity with the Cowork scheduled task before
  retiring it.
- Blog `CLAUDE.md` gotcha or docs entry documenting the roundup path.

## Acceptance criteria

- Topic-coverage inventory committed and explicit about which topic areas
  use envelope-sourced data vs. web-search fallback (if used).
- Monday run produces one `stage:mined` content-queue issue with the
  roundup markdown linked or attached, within 10 minutes of schedule.
- Output quality matches Cowork's output for two consecutive weeks.
- Zero hand-editing of envelope fields required — synthesis is a pure
  function of local index + instruction template + (optional) fallback
  search results.
- Empty-topic failure mode: if a topic area has no items in the window
  and no fallback configured, the roundup says "no significant news this
  week" for that section — no padding, no fabrication.

## Dependencies

- W2 (local envelope index + subscriber populating it).
- **Implicit dependency on NC's source registry covering the roundup's
  topic areas.** Verify before starting; escalate if the gap would expand
  scope materially.

Note: W3 does **not** depend on an NC window-query API — W2's local index
serves window queries. This is per the ADR's "Where synthesis signals
live" section. Do not relitigate.

## Complexity

M with full topic coverage. L if coverage gaps force option (a) (NC
source registry extension). Still M if the gap is narrow enough to pick
option (b) (web-search fallback) — most of the code is the fallback
adapter, not new infrastructure.
