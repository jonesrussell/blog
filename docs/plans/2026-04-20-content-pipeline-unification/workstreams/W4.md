# W4 — Substack as Output Channel

## Objective

Wire Substack as a permanent output channel alongside the existing Buffer
targets (X, LinkedIn, Facebook). Decide between Buffer-integrated posting,
direct-API posting, and manual-paste-with-generated-draft; implement the
chosen path.

## Repos touched

- `~/dev/blog` — content-pipeline skill, channels config, possibly a new
  publishing adapter
- `~/.claude/skills/content-pipeline/` — SKILL.md, channels.env,
  buffer-post.sh or a new adapter script

## Deliverables

- A short decision doc in this workstream's directory capturing:
  - whether Buffer supports Substack as a channel (verify against Buffer's
    current channels list),
  - whether Substack has a usable API or email-to-post path,
  - the chosen path: Buffer / direct / manual-paste-with-draft.
- Implementation of the chosen path. At minimum: draft generation is
  automatic and lands at a known file path or URL. If manual paste is
  required, the distribution path pauses for operator confirmation,
  presents the draft, and transitions the issue to `stage:distributed` on
  confirmation.
- `~/.claude/skills/content-pipeline/SKILL.md` updated with the Substack
  branch of the distribution logic.
- `channels.env` gains a Substack entry (or a clearly documented
  manual-step marker).

## Acceptance criteria

- An issue with `substack` in `suggested_channels[]` produces a Substack
  draft automatically during distribution.
- Distribution does not silently skip Substack. If manual paste is required,
  the pipeline says so and waits for operator confirmation.
- Existing Buffer targets (X, LinkedIn, Facebook) continue to work unchanged.

## Dependencies

None. Runs in parallel with everything else from day one.

## Complexity

M — the uncertainty is in Substack API access, not in wiring.
