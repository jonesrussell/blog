# W2 — Blog Consumer-Enrichment Subscriber

## Objective

Build the blog-side service that subscribes to the NC envelope, maintains a
local envelope index, computes consumer fields, and emits a `stage:mined`
content-queue issue for every signal.

## Repos touched

- `~/dev/blog` — new subscriber service (likely under `tools/` as a Go
  binary; decide during kickoff)

## Deliverables

- Subscriber service reading NC Redis channels. **Default subscription:
  all classifier-published routing-layer channels after W1a deploys.**
  Explicit include/exclude list supported via config.
- Local envelope index keyed on `content_id` with a TTL (default 90 days).
  Implementation: SQLite unless there is a concrete reason against it.
  Stored under `~/.blog-pipeline/` or equivalent; documented in blog
  `CLAUDE.md`.
- Entity → queue-issue index (scans open `content-queue` issues).
- Entity → published-post index (scans `content/posts/**/*.md`).
- NC envelope → content-queue issue translator producing all consumer
  fields listed in `02-contracts.md`. Inference rules are explicit functions
  of envelope fields — deterministic, documented in code and in a short
  blog-side spec.
- Issue writer creating GitHub issues on `jonesrussell/jonesrussell` with
  `content-queue`, `stage:mined`, `source:<origin>`, and appropriate
  `type:*` labels.
- Blog `CLAUDE.md` gains a Gotcha section describing the subscriber, the
  index location, and how to rehydrate from NC Elasticsearch.

## Acceptance criteria

- An NC envelope arriving on a subscribed channel creates exactly one
  `stage:mined` issue with all consumer fields populated.
- Re-delivery of the same `content_id` is idempotent — no duplicate issue.
- `related_queue_issues[]` and `related_published_posts[]` reflect entities
  from the envelope, verified against at least two hand-picked examples.
- Killing the subscriber mid-stream and restarting it does not lose or
  duplicate issues.
- Local index is rehydratable from NC Elasticsearch. Document the recipe.
- Legacy coforge Redis subscriber disabled as part of W2's deploy;
  verified by producing no duplicate issues on `coforge:core` after
  cutover (see Rollout section below).

## Suggested inference rules

These are sketches, not final. The executor refines each rule during
implementation and documents the final version in the subscriber's README.
Divergence from the sketch is expected — but must come with rationale,
not drift.

| Field | Starting rule |
|---|---|
| `suggested_type` | Derive from `content_type` + `content_subtype`. `article` / `blog_post` → blog-post. `article` / `press_release` → text-post. `video` / * → video. `article` / `report` → blog-post. Default: text-post. |
| `suggested_channels[]` | Map from `domain_tags[]` and `topics[]`. `coforge` in domain tags → `{linkedin, x}`. `indigenous` in domain tags → `{linkedin, substack}`. Builder-audience topics (ai, software, devops) with no domain tag → `{x, linkedin}`. Default: `{x, linkedin}`. Multiple domain tags union the channel sets. |
| `format_fit` | `quality_score` plus `quality_components.word_count`. score ≥ 70 AND word_count ≥ 1200 → longform. score ≥ 50 AND word_count < 800 → thread. score ≥ 50 AND word_count 800–1199 → narrative. Anything destined for a roundup intake → roundup-item. |
| `angle_hypothesis` | See "The `angle_hypothesis` decision" below. |
| `audience` | Map from `domain_tags[]`. `indigenous` → indigenous. `coforge` → clients. No domain tag AND `quality_score` ≥ 60 → builders. Everything else → general. |
| `estimated_effort` | `quality_score` bands: ≥ 80 → quick. 40–79 → medium. < 40 → deep. Tempered by `content_type`: video never goes below medium. |
| `related_queue_issues[]` | Entity Jaccard overlap with each open content-queue issue's entity set; include issues above a configurable threshold (default 0.3). |
| `related_published_posts[]` | Same mechanism over the published-post entity index. Include posts above the same threshold. |
| `series_candidate` | Entity-overlap ≥ threshold with any existing series' top entities (via the published-post index, grouped by `series` frontmatter). Null otherwise. First match wins; ties broken by series recency. |
| `voice_notes` | Not auto-derived in v1. Subscriber leaves it empty; production step fills in. |
| `link_anchor_candidates[]` | Top N entity names from `entities[]` that already appear as anchor-eligible phrases in at least two published posts. N = 5. |

All thresholds (0.3, 60, 70, 1200, 800, N=5) are configurable via the
subscriber's config file. The subscriber logs which rule fired for each
field so divergence from the sketch is auditable.

### The `angle_hypothesis` decision

Two positions were considered:

- **A — Templated LLM call.** Populate a short prompt with `entities[]`,
  `topics[]`, `canonical_excerpt`; call a model; cache the result per
  `content_id`. Non-strict determinism: same `content_id` returns the
  cached answer forever; new items hit the model once.
- **B — Mechanical hook template.** Substitute into a fixed string such as
  `"{top_entity} is doing {top_topic}, relevant because {voice_note}"`.
  Leaves creative framing to the human at `stage:curated`.

**Decision: A with caching per `content_id`.** The field's value is its
usefulness, and mechanical templates will not carry that. Caching keeps
replay and backfill deterministic: the first computation writes the cache;
all subsequent deliveries of the same `content_id` return the same string.

Implementation notes:

- Cache lives next to the local envelope index (same store, keyed by
  `content_id`).
- On cache miss, call the model with a short, locked prompt (documented in
  the subscriber's README).
- On model error, fall back to the B-style mechanical template so the
  subscriber never blocks on LLM availability.
- A cache-invalidation CLI exists for regenerating specific entries when
  the prompt evolves.

## Fan-in indexing strategy

`related_queue_issues[]` and `related_published_posts[]` require entity
indexes built over open GitHub issues and the `content/posts/` tree. These
indexes must be:

- **Built once on startup** from a full scan.
- **Refreshed incrementally** as state changes.
- **Never rebuilt from scratch per envelope.** Per-envelope full scans are
  explicitly forbidden — they turn every signal into O(issues + posts) work.

Acceptable refresh strategies (pick one; document the choice in the
subscriber's README):

- Poll the GitHub issue-events API on an interval (e.g., every 5 minutes)
  and apply deltas.
- File-watch `content/posts/` via fsnotify (or equivalent) for post
  additions and removals.
- Rebuild both indexes on subscriber restart. Combined with crash-only
  restart, this is an acceptable minimum — but add incremental refresh
  if restart frequency drops below daily.

The subscriber exposes a health endpoint or log line showing
last-index-refresh timestamps so staleness is visible.

## Rollout

W2's deploy must include **disabling the legacy coforge Redis subscriber
in the same change**. Running W2 and the legacy subscriber in parallel
would produce duplicate content-queue issues on `coforge:core` — one from
each path. This cutover is W2's responsibility, not W5's.

W5's scope stays the broader intake-unification migration (content-mine
workflow rewrite, `source:git` channel wiring, legacy code deletion). The
coforge disable step is simpler because the legacy subscriber consumes the
same channel W2 now owns — disabling it is a one-line config change on
the legacy side.

Deploy checklist:

- Pre-cutover: verify W2 is running and producing `stage:mined` issues for
  a known `coforge:core` item.
- Cutover: disable the legacy coforge subscriber in the same deploy
  change that brings W2 live.
- Post-cutover: verify no duplicate issues are created for the next
  `coforge:core` item.
- Rollback: re-enable the legacy subscriber, pause W2.
